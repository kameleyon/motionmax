# Supabase migrations

Database migrations for MotionMax. Files are applied in **filename order**, so the timestamp prefix is load-bearing — never rename or reorder.

## Filename convention

```
YYYYMMDDHHMMSS_short_description.sql
```

The `YYYYMMDDHHMMSS` prefix is the migration's apply order. The `short_description` is for humans skimming `git log`. Both halves are required.

Older migrations use a `YYYYMMDDHHMMSS_<uuid>.sql` pattern from when the schema was generated by Supabase's web UI. Newer migrations follow the explicit-description convention.

## How to add a migration

1. **Pick a timestamp** strictly greater than the latest existing file. Use UTC. Don't backdate — Supabase's migration runner compares by filename and will refuse to apply if the timestamp is already present in `supabase_migrations.schema_migrations`.
2. **Write the SQL** with `BEGIN; ... COMMIT;` so a partial application can't corrupt schema state.
3. **Make it idempotent** where possible (`CREATE TABLE IF NOT EXISTS`, `ON CONFLICT DO NOTHING`, `CREATE OR REPLACE FUNCTION`). Migrations get re-applied to fresh dev environments and to staging — non-idempotent migrations make those rebuilds painful.
4. **`SECURITY DEFINER` functions** must `SET search_path = public, pg_temp` to prevent search_path injection from the caller's session.
5. **Privilege grants** — `REVOKE ALL ON FUNCTION ... FROM anon;` and `GRANT EXECUTE ... TO authenticated;` for any RPC the client calls. Service-role calls don't need an explicit grant.
6. **Apply via** `npx supabase db push --linked` or `cat <file>.sql | npx supabase db query --linked` for a one-off ad-hoc apply.

## Rollback policy

There is no automated rollback in Supabase migrations. Any migration that drops a column or destroys data should:

1. Be preceded by a database backup (Supabase makes one automatically before each migration apply, but verify it landed).
2. Have an inverse migration ready to apply if production breaks. Don't bury the inverse in a comment — write it as a real `.sql` file in the same commit, named the same plus `_rollback`, and link to it in the forward migration's header.

## Common pitfalls

- **Trailing whitespace** in `pg_get_functiondef` output round-trips can break `CREATE OR REPLACE FUNCTION ... AS $$ ... $$` blocks. When copy-pasting an existing function definition into a new migration, run it through `grep -v '^[[:space:]]*$'` to strip stray blank lines first.
- **`ON CONFLICT DO NOTHING` in an INSERT** silently succeeds even when the seed data was wrong — use `ON CONFLICT (col) DO UPDATE SET ...` if you want the migration to actually update existing rows.
- **`ALTER TABLE ... ADD COLUMN`** is fast on Postgres 11+ when the new column has a constant default and is nullable — but adding NOT NULL with a backfill is a write-locking operation. Big tables (`profiles`, `video_generation_jobs`) need a 3-step pattern: add nullable → backfill in batches → set NOT NULL.
- **Filename collisions across branches** — two devs picking the same timestamp will conflict at merge. The convention is "second commit bumps to the next available second" rather than altering history.

## What lives outside this directory

- **Worker code** that consumes the schema lives in `worker/src/`. Schema-changing migrations should be coordinated with worker deploys to avoid the worker reading a column that was just dropped, or writing to a column that doesn't exist yet.
- **Edge functions** in `supabase/functions/` are deployed independently of migrations. They consume the schema via the Supabase JS client — same coordination concern as worker code.
- **Type generation** — running `npx supabase gen types typescript --linked` after a migration regenerates `src/integrations/supabase/types.ts`. The CI build should fail if types are out of date.
